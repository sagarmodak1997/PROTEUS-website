1. Installation

PROTEUS is a Python package that wraps GPU-accelerated Monte Carlo integration libraries. It auto-detects supported accelerator stacks at launch: NVIDIA CUDA, AMD ROCm/HIP, Apple Silicon Metal/MPS, or CPU-only fallback. The reference validation uses NVIDIA A100 GPUs, but CPU-only installs and other supported accelerator families are handled by the same hardware-detection layer.

1.1 Clone and create the environment

git clone https://github.com/sagarmodak1997/PROTEUS.git
cd PROTEUS
conda create -n proteus python=3.10 -y
conda activate proteus
pip install -r requirements.txt

1.2 GPU backends

At startup, PROTEUS detects the local hardware and chooses the framework device exposed by TensorFlow and PyTorch. Supported accelerator families are:

  • NVIDIA — CUDA via TensorFlow and PyTorch.
  • AMD — ROCm/HIP via tensorflow-rocm and ROCm PyTorch, where available.
  • Apple Silicon — Metal for TensorFlow via tensorflow-metal and MPS for PyTorch.
  • CPU only — automatic fallback when no supported GPU stack is available.

For the reference NVIDIA/CUDA setup, identify your CUDA driver/runtime version with nvidia-smi, then install matching TensorFlow and PyTorch wheels. TensorFlow drives VegasFlow and i-flow; PyTorch drives MadNIS.

# NVIDIA CUDA 12.x
pip install tensorflow[and-cuda]
pip install torch --index-url https://download.pytorch.org/whl/cu121

# NVIDIA CUDA 11.x
pip install tensorflow==2.12
pip install torch --index-url https://download.pytorch.org/whl/cu118

# AMD ROCm/HIP, if supported by your platform
pip install tensorflow-rocm
pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/rocm6.0

# Apple Silicon Metal/MPS
pip install tensorflow-metal
# PyTorch uses MPS when installed on supported Apple Silicon systems

Generic non-NVIDIA, non-ROCm, non-Apple GPU devices are not claimed unless the installed TensorFlow/PyTorch stack exposes them as supported devices.

1.3 Integration libraries

Install the three sampling backends. PROTEUS calls their public APIs directly and never reimplements them, so each must be importable in the environment.

pip install vegasflow
pip install git+https://gitlab.com/i-flow/i-flow.git
pip install git+https://github.com/madnis/madnis.git

For hadronic (proton–proton / Drell-Yan) mode, also install the GPU PDF evaluator and an LHAPDF set manager:

pip install pdfflow lhapdf-management

1.4 Install PROTEUS

Install PROTEUS itself in editable mode from the repository root. This registers the proteus console command and makes import proteus available.

pip install -e .

1.5 Verify your install

Run the built-in environment check. It validates every backend and reports the detected hardware path.

proteus --check

Expected output (yours will differ in versions / GPU count): 

  [OK] Python       3.10.14
  [OK] TensorFlow   2.21.0
  [OK] PyTorch      2.6.0
  [OK] VegasFlow    1.3.0
  [OK] i-flow       found
  [OK] MadNIS       found
  [OK] pdfflow      found
  [OK] GPU          2x NVIDIA A100 80GB PCIe
  All checks passed.

If any line reads [MISSING], install that package (see the exact commands in Troubleshooting). A missing GPU is not fatal: PROTEUS uses CPU execution where the selected backend dependencies support it.

2. Quick Start

The fastest path is the interactive launcher. With the environment active:

conda activate proteus
proteus

This prints the banner and an interactive menu. You can also jump straight to a mode with command-line flags:

proteus              # interactive menu
proteus -t           # terminal interface
proteus -w           # web interface (Gradio, port 8888)
proteus -w -p 9000   # web interface on a custom port
proteus --version    # print version and exit
proteus --check      # validate the environment
proteus --no-banner  # suppress the startup banner

# Equivalent module form (if the console script is not on PATH):
python -m proteus --check

Or run a one-line integration directly in Python:

import numpy as np
import proteus

r = proteus.integrate(
    func    = lambda x: np.exp(-0.5 * np.sum(x**2, axis=-1)),
    n_dim   = 5,
    bounds  = (-4, 4),
    n_calls = 200_000,
)
print(r.summary())
print(r.value, "±", r.error)   # ≈ 98.9  on [-4,4]^5 (backend: vegasflow)

3. Python API

Everything lives under the proteus namespace. For most tasks the top-level helpers are all you need; the advanced classes give finer control.

3.1 Top-level helpers

The package exposes a small, stable public API:

import proteus

# General-purpose integration — backend auto-selected by the dispatcher
r = proteus.integrate(func=my_func, n_dim=5, bounds=(-4, 4), n_calls=200_000)

# A HEP cross section in one call (e+e- → μ+μ- at the Z pole)
s = proteus.compute_cross_section(process='ee_to_mumu', ecm=91.2, n_calls=500_000)

proteus.load_config()   # load the default configuration dict
proteus.cite()          # print the BibTeX entry

Both integrate and compute_cross_section return an IntegrationResult. Useful attributes:

r.value                     # central estimate
r.error                     # Monte Carlo (statistical) error
r.backend                   # which backend ran ('vegasflow', 'iflow', 'madnis')
r.relative_error_percent    # error as a percentage of the value
print(r.summary())          # formatted multi-line report

3.2 General integration

Integrate any vectorised NumPy function. The input x has shape (batch, n_dim); the function must return a 1-D array of length batch. This default compatibility path may cross a Python/NumPy callback bridge; it is general and convenient, but it is not the path used for GPU-speedup benchmark claims.

import numpy as np
import proteus

result = proteus.integrate(
    func    = lambda x: np.exp(-0.5 * np.sum(x**2, axis=-1)),
    n_dim   = 5,
    bounds  = (-4, 4),
    n_calls = 200_000,
)
print(result.value, "±", result.error)   # ≈ 98.9 ± 0.1  (backend: vegasflow)

3.3 TensorFlow-native expression mode

For performance benchmarks with expression strings on the VegasFlow path, PROTEUS provides an opt-in TensorFlow-native mode. This avoids the Python/NumPy callback bridge and lets VegasFlow evaluate the expression as TensorFlow tensor code.

result = proteus.integrate(
    func           = "exp(-0.5 * sum(x**2, axis=1))",
    n_dim          = 4,
    bounds         = (-6.0, 6.0),
    n_calls        = 10_000_000,
    peaked         = False,
    backend        = "vegasflow",
    integrand_mode = "tensorflow",
)

Use integrand_mode="numpy" for the default compatibility path. TensorFlow-native mode currently applies to expression strings routed through VegasFlow; arbitrary Python callables, file-based matrix elements, i-flow, and MadNIS retain their existing compatibility behavior.

3.4 HEP cross sections

For full control over physics processes, use PhysicsFrontend:

from proteus.frontends.physics_frontend import PhysicsFrontend

pf = PhysicsFrontend(verbose=True)

# e+e- → μ+μ- at the Z pole
r = pf.compute_cross_section(process='ee_to_mumu', ecm=91.2, n_calls=500_000)
print(f"σ = {r.value:.1f} ± {r.error:.1f} pb")   # ≈ 2010 ± 1 pb

# Drell-Yan + Z' at 200 GeV
r = pf.compute_cross_section(
    process='drell_yan_zprime', ecm=200.0, n_calls=500_000,
    quark_pdg_id=2, zprime_mass=150.0, zprime_width=1.5,
)

3.5 Hadronic (pp) cross sections

Convolve a partonic process with PDFs via PDFFlow. Requires the pdfflow and lhapdf-management install from step 1.3. PROTEUS sums over up- and down-type quark flavours automatically.

from proteus.frontends.physics_frontend import PhysicsFrontend

pf = PhysicsFrontend(verbose=True)

# Drell-Yan at LHC 13 TeV, in the Z window 60 < M_ll < 120 GeV
r = pf.compute_cross_section_pp(
    process='drell_yan', sqrt_s=13000.0,
    mll_min=60.0, mll_max=120.0,     # dilepton mass window (see note)
    n_calls=2_000_000,
)
print(r.value, "pb")               # ~2000 pb (single lepton flavour, NNPDF3.1 NLO)

Always set a dilepton mass window for Drell-Yan. The γ* contribution has a 1/ŝ pole as Mℓℓ→0, so the inclusive (no-cut) integral is dominated by the unphysical low-mass region and does not converge. With mll_min/mll_max the result is physical and validated against MadGraph to 0.7% (matching PDF, 60–120 GeV window: PROTEUS 1463±6 pb vs MadGraph 1453±2 pb).

3.6 SMEFT operators

Scan dimension-6 Warsaw-basis Wilson coefficients with EFTFrontend. At every coefficient set to zero, the result reduces to the Standard Model.

from proteus.frontends.eft_frontend import EFTFrontend

eft = EFTFrontend(verbose=False)
r = eft.compute_cross_section(
    process='ee_to_mumu_eft', ecm=200.0,
    wc_dict={'ll_1221': 0.01}, Lambda=1000.0, n_calls=500_000,
)

3.7 Differentiable cross sections

PROTEUS computes dσ/dθ — the gradient of a cross section with respect to a physics parameter — via the Leibniz rule (the derivative is itself a Monte Carlo integral).

grads = pf.compute_cross_section_gradient(
    process='ee_to_mumu', ecm=200.0,
    wrt={'mz': (91.1876, 0.1)},
    n_calls=500_000,
)
print(grads['mz'].value, "pb/GeV")

3.8 Custom matrix elements

Load a squared matrix element from a string expression, a file, or a Python callable:

from proteus.physics_utils.matrix_loader import load_matrix_element

m2 = load_matrix_element("x^2 + y^2")   # also accepts a file path or a callable

4. Web Interface

Launch the Gradio interface and open the URL it prints (default port 8888):

proteus -w
# → http://localhost:8888

proteus -w -p 9000   # choose a different port

Seven tabs cover the most common phenomenology workflows:

  • ⚙ General Integration — any function, 1 – 50 dimensions
  • ⚛ HEP Physics (e⁺e⁻) — Drell-Yan, ee→μμ, Z′
  • 🔬 Hadronic (pp) — LHC / Tevatron with PDFFlow (set a dilepton mass window)
  • 📐 Phase Space Volume — RAMBO with analytic comparison
  • 🧪 EFT / SMEFT — Warsaw-basis Wilson coefficient scans
  • ∂ Gradient dσ/dθ — differentiable cross sections
  • 📦 ONNX Export — i-flow / MadNIS to ONNX (best-effort)

5. Backend Dispatcher

PROTEUS routes each integrand to a backend using dimension, peakedness, and any explicit backend override. The current default policy is: D ≤ 4 goes to VegasFlow; smooth D = 5-12 goes to VegasFlow; peaked or resonant D = 5-12 goes to MadNIS; and D > 12 goes to i-flow. A Vegas-grid concentration diagnostic is used only where it is reliable.

ConditionBackend
D ≤ 4VegasFlow (TensorFlow)
D = 5-12, smoothVegasFlow (TensorFlow)
D = 5-12, peaked / resonantMadNIS (PyTorch)
D > 12i-flow (TensorFlow)
backend=...Explicit user override

5.1 Validated GPU performance scope

GPU execution has been validated across representative backend routes. Speedup has been benchmarked for supported native paths, especially TensorFlow-native VegasFlow expression workloads at large call counts. Arbitrary Python/NumPy callback integrands are supported for compatibility, but they should not be described as fully GPU-native or guaranteed faster than CPU.

You normally never choose a backend by hand — but if you want to, see the backend FAQ entry in Troubleshooting.

6. Troubleshooting / FAQ

proteus: command not found

The console script is registered when you run pip install -e . inside the active environment. If the command is not found:

  • Make sure the environment is active: conda activate proteus.
  • Reinstall from the repository root: pip install -e ..
  • As a fallback you can always use the module form: python -m proteus --check.

--check reports a missing backend

Install whichever package is flagged [MISSING]:

pip install vegasflow                                  # VegasFlow
pip install git+https://gitlab.com/i-flow/i-flow.git   # i-flow
pip install git+https://github.com/madnis/madnis.git   # MadNIS
pip install pdfflow lhapdf-management                  # hadronic (pp) mode

Then re-run proteus --check to confirm all lines read [OK].

GPU not detected

If proteus --check shows no GPU, PROTEUS can use CPU execution where the selected backend dependencies support it. CPU fallback is useful for testing and small jobs, but the reference performance validation is GPU-focused. Common causes of a missing GPU:

  • NVIDIA: TensorFlow/PyTorch wheels do not match your CUDA version — check nvidia-smi and reinstall matching wheels.
  • AMD: ROCm is missing or the Python environment lacks tensorflow-rocm / ROCm PyTorch.
  • Apple Silicon: tensorflow-metal is missing for TensorFlow-backed VegasFlow/i-flow; PyTorch should expose MPS for MadNIS.
  • No supported accelerator runtime is installed on the host.

What happens at launch?

The launcher runs the hardware detector before starting the terminal or web UI. It reports whether the system is using NVIDIA CUDA, AMD ROCm/HIP, Apple Metal/MPS, or CPU. On AMD and Apple systems it can point out missing optional GPU packages; on multi-GPU systems it offers per-backend GPU assignment. If no supported GPU stack is available, workers use CPU-compatible backend paths where available.

Can I import TensorFlow and PyTorch in the same script?

Yes. PROTEUS runs each backend in an isolated worker subprocess, so TensorFlow and PyTorch never share a Python process at runtime. Importing both proteus and any other framework in your own code is completely fine — there are no in-process conflicts.

My hadronic (pp) Drell-Yan cross section looks huge or unstable

You almost certainly forgot the dilepton mass window. Drell-Yan via γ* has a 1/ŝ pole as Mℓℓ→0, so the inclusive (no-cut) cross section is dominated by the unphysical low-mass region and does not converge — a handful of near-pole points swamp the estimate. Pass mll_min and mll_max (e.g. 60 and 120 GeV around the Z), exactly as every real measurement does. With a window the result is physical, stable, and validated against MadGraph to 0.7% (60–120 GeV, matching PDF). PROTEUS sums over up- and down-type quark flavours automatically.

ONNX export returns onnx_valid=False

ONNX export is best-effort. Some flow architectures use operations that tf2onnx cannot convert, and in a subprocess GPU context onnxruntime may fail to create a session. When that happens the export still writes the model files but reports onnx_valid=False. The native checkpoint (TensorFlow / PyTorch) remains the reliable artifact.

How do I choose the right backend?

You usually don't — the dispatcher picks for you based on dimensionality and a peakedness diagnostic. As a rule of thumb: D ≤ 4 → VegasFlow; smooth D = 5-12 → VegasFlow; peaked or resonant D = 5-12 → MadNIS; D>12 → i-flow. If you need a specific backend, pass backend="vegasflow", backend="iflow", or backend="madnis".
